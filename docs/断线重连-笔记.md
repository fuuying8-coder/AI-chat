# SSE 流式聊天：断线重连设计笔记（前端实现）

> 适用：本项目前端通过 `fetch + response.body.getReader()` 读取后端 `text/event-stream`（SSE）流。
> 目标：网络中断/服务重启/连接被重置时，自动重连并尽量避免重复内容。

## 1. 关键结论（你问的 3 个点）

### 1) “续写”是否意味着重新请求大模型？
是的。

- 断线后无法“继续读同一个 HTTP 流”，只能 **重新发起一个新的 HTTP 请求**。
- 在本项目里，重连由前端循环调用 `createResponse()` 实现：每次重试都会再次调用 `createChatCompletion` / `createRagChatCompletion` / `createCaseStudyChatCompletion`，从而触发后端重新请求上游/LLM。

入口见：
- `createResponse` 构造（重试时再次调用 API）：src/views/ChatView.vue#L149

### 2) 如何判断“断线”需要重连？
本实现把“断线/可重试的异常”定义为：**在读取流或解析 SSE 时抛出异常**（且不是用户主动取消）。

判断点：
- 读取流的过程中抛异常（常见：`TypeError: Failed to fetch`、connection reset、stream terminated 等）
- SSE 行里出现 `data: {"error": ...}`（后端主动发送的错误事件）
- 用户点击“停止”触发 `AbortController.abort()`：这不是断线，不重试

核心判断代码：
- 可重试异常识别：src/utils/messageHandler.js#L30
- abort 异常识别：src/utils/messageHandler.js#L25

> 注意：如果连接“正常结束”但没有 `[DONE]`，当前实现不会强行判定为断线（因为 `reader.read()` 只会返回 `done=true`）。

### 3) 断线重连是如何实现的？
核心思路：

1. 把“发起请求”抽象成 `createResponse()` 回调（每次重试都能重新 fetch）
2. 用 `for attempt in 1..N` 包裹整个流消费逻辑
3. 发生异常时：
   - 如果是 `AbortError`：直接抛出，不重试
   - 如果是“可重试异常”：指数退避 sleep 后重试
4. 重连后避免重复：
   - 普通对话：可选择“提示模型从末尾续写”（通过追加一条 user 指令）
   - RAG / CaseStudy：不能追加 user（会覆盖原问题），改用“前缀去重”跳过重复 token

主循环：src/utils/messageHandler.js#L190

### 2.2 messageHandler：重连主循环（指数退避 + 抖动）
文件：src/utils/messageHandler.js

- 重连入口：src/utils/messageHandler.js#L190
- 退避计算：src/utils/messageHandler.js#L19
- sleep：src/utils/messageHandler.js#L14

伪代码：
```js
for attempt in 1..maxRetries+1:
  try:
    response = await createResponse({attempt, resume})  // 这里会重新 fetch
    await consumeStream(response)                       // 读 SSE 并 update UI
    return                                              // 正常结束
  catch (err):
    if AbortError: throw
    if not retryable or lastAttempt: throw
    await sleep(backoffMs(attempt))
```

### 2.3 “续写提示”如何构造（仅普通对话使用）
文件：src/utils/messageHandler.js

- `_buildResumeMessages`：src/utils/messageHandler.js#L51

作用：把已输出内容塞回历史，追加一条 user：
- “请从已输出内容末尾继续，不要重复”
- 同时提供“末尾片段”帮助对齐

### 2.4 重连去重：前缀去重（避免重发导致 UI 叠加重复）
文件：src/utils/messageHandler.js

- `dedupeActive` 开关：src/utils/messageHandler.js#L216

机制：
- attempt > 1 时开启去重
- 把新流来的 `deltaContent` 与已输出 `accumulatedContent` 做逐字符对齐
- 对齐阶段：跳过相同字符
- 一旦发现不同：停止去重，开始正常追加

> 这个策略特别适合 RAG/CaseStudy：它们重连时重发“同一问题”，模型常从头复述，前缀去重能跳过重复前缀。

## 3. 为什么 RAG / CaseStudy 不能用“追加一条 user 让模型继续”？

因为后端实现（Python）是这么取问题的：
- RAG：取 messages 里最后一条 `role=user` 当 question
- CaseStudy：同样取最后一条 `role=user` 做意图解析

如果你重连时在 messages 末尾追加 `role=user` 的“请继续”，后端就会把它当成新问题，导致检索/意图解析跑偏。

因此：
- RAG/CaseStudy：重连必须“原请求重发”，靠前缀去重/或未来服务端支持断点续传
- 普通 qwen：可以用“续写提示”降低重复概率

## 4. 可调参数（当前默认值）

在 src/views/ChatView.vue#L179 调整：
- `maxRetries`: 3
- `baseDelayMs`: 600
- `maxDelayMs`: 4000
- `jitterRatio`: 0.2

建议：
- 内网/本地开发：`maxRetries=1~2` 即可
- 生产环境：可提高到 `3~5`，并把 `maxDelayMs` 拉到 `8000~15000`

---

## 5. 局限 & 可选增强（不影响当前实现）

- 无“硬判定 done=true 但没 [DONE]”的场景：需要你在协议层保证 `[DONE]` 或服务端发心跳。
- 真正的断点续传：需要服务端支持（例如传 offset/token id），仅前端无法做到绝对精确。
- 目前 `speed/tokens` 只是 UI 估算；跨重连的统计可改成累计。
