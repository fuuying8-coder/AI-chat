# LLM Chat 项目总结：前后端职责与数据流

本文档说明前端、后端分别做了什么，关键代码位置，以及数据如何流动。

---

## 一、整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│  前端 (Vue 3 + Vite)                                              │
│  - ChatView / ChatInput / ChatMessage                             │
│  - api.js（请求封装）                                              │
│  - upload 模块（切片、并发池、断点续传）                             │
│  - messageHandler（流式/非流式响应处理）                            │
└───────────────────────────┬───────────────────────────────────────┘
                            │ HTTP (localhost:3001)
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│  后端 (FastAPI, server_python/main.py)                            │
│  - /api/test, /api/chat/qwen                                      │
│  - /api/files/upload, /api/upload/check|chunk|complete|status      │
│  - 千问调用：LangChain / OpenAI 兼容接口 + 百炼                     │
└───────────────────────────┬───────────────────────────────────────┘
                            │
                            ▼
                    阿里云百炼 (DashScope)
```

## 二、前端做了什么

### 2.1 聊天页与发送流程

**职责**：用户输入 → 可选长文档上传 → 调用千问接口 → 流式/非流式展示回复，支持停止生成与自动滚到底部。

**关键文件**：`src/views/ChatView.vue`、`src/components/ChatInput.vue`、`src/utils/api.js`、`src/utils/messageHandler.js`。

**发送流程（结合代码）**：

1. **用户点击发送**  
   `ChatInput` 触发 `send`，带上 `{ text, files }`。

2. **ChatView 处理发送**（`ChatView.vue` 约 67–130 行）  
   - 先往 store 里追加一条用户消息和一条空的助手消息。  
   - 若有附件且是长文档（docx/pdf/txt/md），先调用 `uploadFileForExtract(docFile.raw)` 拿到 `file_id`。  
   - 设置 `isLoading = true` 并立刻 `scrollToBottom()`，让新消息出现在可视区。  
   - 创建 `AbortController`，用其 `signal` 调用 `createChatCompletion(messages, { fileId, stream, signal })`。  
   - 根据返回是否为流式（`response.body.getReader`）走 `messageHandler.handleResponse(response, isStream, updateCallback)`。

```javascript
// ChatView.vue 片段：发送 + 流式回调里滚到底部
abortController = new AbortController()
const response = await createChatCompletion(messages, {
  fileId: fileId || undefined,
  stream: useStream,
  signal: abortController.signal,
})
await messageHandler.handleResponse(response, isStreamResponse, (content, ...) => {
  chatStore.updateLastMessage(content, ...)
  scrollToBottom()  // 每收到一块内容就滚到底部
})
```

3. **停止生成**  
   用户点击「停止」时触发 `handleStop()`，执行 `abortController.abort()`；`fetch` 被取消，流式读取会抛出 `AbortError`，前端不把最后一条消息改成错误文案，只保留已生成内容。

4. **发送按钮状态**（`ChatInput.vue`）  
   - `loading=false`：蓝色发送按钮，点击发送。  
   - `loading=true`：红色「停止」+ 旋转图标，点击触发 `stop` 事件，由 ChatView 调用 `handleStop()`。

---

### 2.2 请求封装与流式/非流式

**职责**：统一请求后端千问接口，支持 `file_id`、`stream`、`signal`（取消），流式时返回原始 `Response` 供消费 SSE。

**关键文件**：`src/utils/api.js`。

**关键逻辑**：

- `createChatCompletion(messages, extraOptions)`：  
  - `extraOptions`: `fileId`、`stream`（默认从 `settingStore.settings.stream` 取）、`signal`（AbortController.signal）。  
  - 请求体里带 `stream: useStream`，若传了 `signal` 则 `fetch(..., { signal })`。  
  - **流式**：不读 body，直接 `return response`，由上层用 `response.body.getReader()` 读 SSE。  
  - **非流式**：`await response.json()` 后返回 `{ choices, usage, speed }`。

```javascript
// api.js：流式时返回 Response，非流式时返回解析后的 JSON
if (useStream) {
  return response
}
const data = await response.json()
return data
```

- `uploadFileForExtract(file, onProgress)`：  
  - 若 `file.size > CHUNKED_UPLOAD_THRESHOLD`（2MB），走 `runUploadTask`（分片+断点续传），否则走单文件 `POST /api/files/upload`。  
  - 最终都返回 `{ file_id, filename }`，供聊天时传 `fileId`。

---

### 2.3 流式响应解析

**职责**：消费 SSE 流，按行解析 `data: {...}`，累积 `content`/`reasoning_content`，每收到一块就调用 `updateCallback`，用于打字机效果。

**关键文件**：`src/utils/messageHandler.js`。

**关键逻辑**：

- `handleStreamResponse(response, updateCallback)`：  
  - 用 `response.body.getReader()` + `TextDecoder` 读流。  
  - 按 `\n` 拆行，识别以 `data: ` 开头的行；若为 `data: [DONE]` 则结束。  
  - 否则 `JSON.parse` 取 `data.choices[0].delta.content`（及 `reasoning_content`），累加后调用 `updateCallback(accumulatedContent, accumulatedReasoning, tokens, speed)`。  
  - 若解析到 `data.error`，则设 `hasError` 并抛出。

```javascript
// messageHandler.js：流式解析
if (data.choices?.[0]?.delta) {
  const content = data.choices[0].delta.content || ''
  accumulatedContent += content
  updateCallback(accumulatedContent, accumulatedReasoning, ...)
}
```

- `handleNormalResponse(response, updateCallback)`：非流式时从 `response.choices[0].message` 取 `content` 等，调用一次 `updateCallback`。

---

### 2.4 大文件上传（Upload Task 模块）

**职责**：文件切片、内容哈希、并发池上传、断点续传、状态持久化，与后端 check/chunk/complete/status 协议配合。

**关键文件**：`src/upload/slicer.js`、`src/upload/concurrencyPool.js`、`src/upload/uploadState.js`、`src/upload/uploadTask.js`。

**数据流（与后端协作）**：

1. **切片与哈希**（`slicer.js`）  
   固定 2MB 一块，每块算 SHA-256；文件级 hash = SHA-256(concat(chunk_hashes))，用于去重和标识。

2. **询问已存在**（`uploadTask.js`）  
   `POST /api/upload/check`，body: `{ file_hash, file_name, total_chunks, chunk_size }`。  
   后端返回 `{ exists?, file_id?, uploaded_chunks: number[] }`。若 `exists && file_id` 直接返回，不再上传。

3. **只传缺失块**（`uploadTask.js` + `concurrencyPool.js`）  
   用并发池（默认 4）只上传 `uploaded_chunks` 中不存在的 index；每块 `POST /api/upload/chunk`，form: `file_hash, chunk_index, total_chunks, file_name, chunk`(文件)。

4. **通知完成并轮询**（`uploadTask.js`）  
   `POST /api/upload/complete`，body 同 check。后端返回 `job_id`。  
   前端轮询 `GET /api/upload/status?job_id=xxx`，直到 `status === 'success'` 拿到 `file_id`，或 `failed` 抛错。

5. **状态持久化**（`uploadState.js`）  
   `localStorage` 按 `fileHash` 存 `{ status, uploadedChunks, jobId, fileId, ... }`，刷新后可恢复进度。

## 三、后端做了什么

### 3.1 基础与健康检查

**关键文件**：`server_python/main.py`。

- **CORS**：允许前端（如 5173）访问。  
- **GET /health**：返回服务状态。  
- **GET /api/test**、**GET /test**：测试连通性，返回 `{ success, message, backendUrl }`。  
- **模型归一化**：`_normalize_qwen_model(model)`：若请求里带了非 qwen 模型（如 DeepSeek），自动改为 `qwen-max`，避免百炼 404。

---

### 3.2 千问聊天接口：`POST /api/chat/qwen`

**职责**：接收前端消息列表和可选 `file_id`，调用阿里云百炼（千问），支持流式（SSE）与非流式（JSON）。

**请求体**（前端发来的典型结构）：

```json
{
  "model": "qwen-max",
  "messages": [ { "role": "user", "content": "你好" } ],
  "stream": true,
  "file_id": "可选，来自上传或分片合并"
}
```

**后端逻辑概览**：

1. **解析**  
   `body`: `messages`, `model`, `file_id`, `stream`（默认 True）。

2. **有 file_id（长文档）**  
   - 使用 `qwen-long`，在首条前插入或合并 system：`content = "fileid://" + file_id`。  
   - **流式**：用百炼 `client.chat.completions.create(..., stream=True)`，对每个 chunk 的 `delta.content` 封装成 SSE：`data: {"choices":[{"delta":{"content":"..."}}]}\n\n`，最后发 `data: [DONE]\n\n`，通过 `StreamingResponse` 返回。  
   - **非流式**：同一 `create(..., stream=False)`，取 `choices[0].message.content`，返回 JSON：`{ choices, usage, speed }`。

3. **无 file_id（普通对话）**  
   - **流式**：`call_qwen_stream_langchain(messages, model)`，内部用 LangChain `ChatOpenAI` 的 `astream`，每块转成同上 SSE 格式，再 `StreamingResponse`。  
   - **非流式**：`call_qwen_with_langchain(messages, model)`，`ainvoke` 取整段 `response.content`，返回 JSON。

4. **SSE 封装**（`_sse_data`）  
   将 Python 对象或字符串转为 `data: {...}\n\n` 的 bytes，保证前端能按行解析。

```python
# main.py：流式返回 SSE
def _sse_data(obj) -> bytes:
    payload = json.dumps(obj, ensure_ascii=False) if not isinstance(obj, str) else obj
    return f"data: {payload}\n\n".encode("utf-8")

# 无 file_id 时流式
return StreamingResponse(
    call_qwen_stream_langchain(messages, model=model),
    media_type="text/event-stream",
    headers={"Cache-Control": "no-cache", "Connection": "keep-alive"},
)
```

---

### 3.3 单文件上传：`POST /api/files/upload`

**职责**：接收一个文件，写入临时路径，调用百炼 `files.create(file=..., purpose="file-extract")`，返回 `{ file_id, filename }`，供聊天时作为 `file_id` 使用。

**数据流**：前端 FormData 只传一个 `file` → 后端读入 → 临时文件 → 百炼 SDK → 返回 `file_id`。

---

### 3.4 分片上传：check / chunk / complete / status

**职责**：按 `file_hash` 分目录存 chunk，去重，异步合并后上传百炼并写 registry，前端可断点续传与轮询合并结果。

**目录与结构**：

- `uploads/chunks/{file_hash}/0`, `1`, `2`, ...：每个分片一个文件。  
- `uploads/merged/`：合并时的临时文件（合并完即删）。  
- `uploads/merged_registry.json`：`file_hash -> file_id`，重启后仍可识别已合并文件。

**接口与数据流**：

| 接口 | 方法 | 前端 → 后端 | 后端 → 前端 |
|------|------|-------------|-------------|
| **/api/upload/check** | POST | `file_hash, file_name, total_chunks, chunk_size` | `exists?, file_id?, uploaded_chunks[]` |
| **/api/upload/chunk** | POST | form: `file_hash, chunk_index, total_chunks, file_name, chunk`(文件) | `{ ok: true }` |
| **/api/upload/complete** | POST | 同 check | `{ job_id }` |
| **/api/upload/status** | GET | `job_id` | `{ status, file_id?, error? }` |

**合并与去重**（`_do_merge`）：

- 后台异步：按 index 顺序读 chunk 写入一个合并文件，再调用百炼 `files.create` 得到 `file_id`。  
- 成功后将 `file_hash -> file_id` 写入 `merged_registry.json`，删除该 `file_hash` 的 chunk 目录和临时合并文件。  
- **check** 时若在 `merge_jobs` 或 `merged_registry.json` 中已有该 `file_hash`，直接返回 `exists: true, file_id`，避免重复上传。

**清理**：启动时 `_cleanup_stale_chunks()` 删除超过 `CHUNK_EXPIRE_HOURS`（默认 24h）未合并的 chunk 目录。

## 四、端到端数据流小结

### 4.1 普通聊天（无附件、流式）

1. 用户输入 → ChatInput 发出 `send`。  
2. ChatView：追加 user + 空 assistant，`setIsLoading(true)`，`scrollToBottom()`，`createChatCompletion(messages, { stream: true, signal })`。  
3. api.js：`POST /api/chat/qwen`，body 含 `messages, stream: true`，带 `signal`；返回 `Response` 不读 body。  
4. 后端：`stream=true` 且无 `file_id` → `StreamingResponse(call_qwen_stream_langchain(...))`，SSE 每块 `data: {"choices":[{"delta":{"content":"..."}}]}\n\n`。  
5. 前端：`messageHandler.handleStreamResponse(response, updateCallback)`，reader 读流 → 按行解析 → 累加 content → 每次 `updateCallback` → `chatStore.updateLastMessage` + `scrollToBottom()`。  
6. 用户可点「停止」→ `abortController.abort()` → fetch 取消 → 流中断，保留已生成内容。

### 4.2 带长文档的聊天（先上传再流式）

1. 用户选 docx/pdf 等并发送。  
2. ChatView：若有长文档附件，先 `uploadFileForExtract(docFile.raw)`：  
   - 小文件：`POST /api/files/upload` → `file_id`。  
   - 大文件：runUploadTask → check → 传缺失 chunk → complete → 轮询 status → 得到 `file_id`。  
3. `createChatCompletion(messages, { fileId, stream: true, signal })`。  
4. 后端：有 `file_id` → system 注入 `fileid://{file_id}`，用 `qwen-long` 流式调用百炼，同样 SSE 输出。  
5. 前端：与 4.1 相同，流式解析 + 更新最后一条消息 + 滚到底部。

### 4.3 大文件分片上传到拿到 file_id

1. 前端：slicer 切片并算 file_hash、chunk hashes。  
2. `POST /api/upload/check` → 后端查 chunk 目录与 registry，返回 `uploaded_chunks` 或 `file_id`。  
3. 前端只上传缺失 index 的 chunk（并发池），每块 `POST /api/upload/chunk`。  
4. 全部传完 `POST /api/upload/complete` → 后端创建合并任务，返回 `job_id`。  
5. 前端轮询 `GET /api/upload/status?job_id=xxx`，直到 `status === 'success'`，拿到 `file_id`。  
6. 该 `file_id` 再进入 4.2 的聊天请求体，用于 qwen-long 长文档解析。
