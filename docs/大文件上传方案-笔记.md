# 大文件上传方案笔记

> 能力：切片上传、断点续传、秒传、上传进度持久化至 localStorage，刷新后可恢复任务。

---

## 一、整体流程

```
用户选择文件
    → 前端：切片 + 计算文件 hash（SHA-256）
    → 调用 /api/upload/check（带 file_hash）
        → 若服务端已有该文件：返回 exists + file_id → 秒传结束
        → 否则返回已存在的 chunk 下标 uploaded_chunks
    → 只上传「未上传」的 chunk（并发池限制并发数）
    → 每上传完一个 chunk，把进度写入 localStorage
    → 全部 chunk 上传完 → 调用 /api/upload/complete
    → 后端异步合并，返回 job_id
    → 前端轮询 /api/upload/status?job_id=xxx 直到 success，拿到 file_id
    → 更新 localStorage 为 success + file_id，供刷新后恢复/秒传
```

---

## 二、前端模块划分

| 模块 | 职责 |
|------|------|
| **slicer.js** | 文件切片、每块 SHA-256、文件整体 hash（用于秒传/断点标识） |
| **concurrencyPool.js** | 并发池，限制同时上传的 chunk 数量 |
| **uploadState.js** | 任务状态读写 localStorage，key: `upload_task_{fileHash}` |
| **uploadTask.js** | 串联：切片 → check → 只传缺失 chunk → complete → 轮询 status，并持久化状态 |

## 三、前端核心代码

### 3.1 切片与文件 Hash（slicer.js）

- 固定块大小（如 2MB）切片，每块计算 SHA-256。
- 文件唯一 hash = `SHA256(concat(chunk_hashes))`，用于秒传与断点标识。

```javascript
const DEFAULT_CHUNK_SIZE = 2 * 1024 * 1024 // 2MB

export async function hashBlob(blob) {
  const buf = await blob.arrayBuffer()
  const hashBuffer = await crypto.subtle.digest('SHA-256', buf)
  return Array.from(new Uint8Array(hashBuffer))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('')
}

export async function sliceAndHash(file, options = {}) {
  const chunkSize = options.chunkSize ?? DEFAULT_CHUNK_SIZE
  const totalChunks = Math.ceil(file.size / chunkSize)
  const chunks = []

  for (let i = 0; i < totalChunks; i++) {
    const start = i * chunkSize
    const end = Math.min(start + chunkSize, file.size)
    const blob = file.slice(start, end)
    const hash = await hashBlob(blob)
    chunks.push({ index: i, blob, hash })
  }

  const concatHashes = chunks.map((c) => c.hash).join('')
  const concatBuf = new TextEncoder().encode(concatHashes)
  const fileHashBuffer = await crypto.subtle.digest('SHA-256', concatBuf)
  const fileHash = Array.from(new Uint8Array(fileHashBuffer))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('')

  return { fileHash, chunks, totalChunks, fileName: file.name, fileSize: file.size, chunkSize }
}
```

### 3.2 并发池（concurrencyPool.js）

- 限制同时执行的 Promise 数量，避免浏览器并发过高。

```javascript
export function createConcurrencyPool(concurrency = 4) {
  let running = 0
  const queue = []

  function runNext() {
    if (running >= concurrency || queue.length === 0) return
    const { task, resolve, reject } = queue.shift()
    running++
    Promise.resolve(task())
      .then(resolve, reject)
      .finally(() => {
        running--
        runNext()
      })
  }

  function run(task) {
    return new Promise((resolve, reject) => {
      queue.push({ task, resolve, reject })
      runNext()
    })
  }
  return run
}
```

### 3.3 状态持久化（uploadState.js）

- Key：`upload_task_{fileHash}`。
- 状态：`waiting` | `uploading` | `success` | `failed`。
- 持久化字段示例：`status`、`fileHash`、`fileName`、`totalChunks`、`uploadedChunks`、`jobId`、`fileId`、`error`。

```javascript
const STORAGE_KEY_PREFIX = 'upload_task_'

export const UploadStatus = {
  WAITING: 'waiting',
  UPLOADING: 'uploading',
  SUCCESS: 'success',
  FAILED: 'failed',
}

function storageKey(fileHash) {
  return `${STORAGE_KEY_PREFIX}${fileHash}`
}

export function getTaskState(fileHash) {
  try {
    const raw = localStorage.getItem(storageKey(fileHash))
    return raw ? JSON.parse(raw) : null
  } catch {
    return null
  }
}

export function setTaskState(fileHash, state) {
  try {
    localStorage.setItem(storageKey(fileHash), JSON.stringify(state))
  } catch (e) {
    console.warn('upload state persist failed', e)
  }
}

export function clearTaskState(fileHash) {
  try {
    localStorage.removeItem(storageKey(fileHash))
  } catch {}
}

/** 列出所有未完成任务，用于刷新后恢复列表 */
export function listPendingTasks() {
  const list = []
  try {
    for (let i = 0; i < localStorage.length; i++) {
      const key = localStorage.key(i)
      if (key && key.startsWith(STORAGE_KEY_PREFIX)) {
        const raw = localStorage.getItem(key)
        if (raw) {
          const state = JSON.parse(raw)
          if (state.status === UploadStatus.WAITING || state.status === UploadStatus.UPLOADING) {
            list.push(state)
          }
        }
      }
    }
  } catch {}
  return list
}
```

### 3.4 上传任务主流程（uploadTask.js）要点

1. **切片 + hash**：`sliceAndHash(file, { chunkSize })`。
2. **读持久化**：若已有 `success` 且带 `fileId`，直接返回（秒传/恢复）。
3. **check 接口**：`POST /api/upload/check`，传 `file_hash`、`file_name`、`total_chunks`、`chunk_size`。  
   - 若返回 `exists: true` 且带 `file_id` → 秒传，写 localStorage 后返回。  
   - 否则用返回的 `uploaded_chunks` 得到「未上传」的 chunk 列表。
4. **只传缺失 chunk**：用并发池 `run(uploadTask(chunk))`，每完成一个就 `setTaskState(fileHash, { ...state, uploadedChunks })`，实现断点续传 + 进度持久化。
5. **complete**：`POST /api/upload/complete`，拿 `job_id`，写入 state。
6. **轮询 status**：`GET /api/upload/status?job_id=xxx`，直到 `success` 拿到 `file_id`，再次 `setTaskState` 为 success；失败则写 `failed` + `error`。

单 chunk 上传示例：

```javascript
async function uploadOneChunk(backendBaseUrl, getAuthHeader, fileHash, fileName, totalChunks, chunk) {
  const form = new FormData()
  form.append('file_hash', fileHash)
  form.append('chunk_index', String(chunk.index))
  form.append('total_chunks', String(totalChunks))
  form.append('file_name', fileName)
  form.append('chunk', chunk.blob)
  const res = await fetch(`${backendBaseUrl}/api/upload/chunk`, {
    method: 'POST',
    headers: getAuthHeader() ? { Authorization: getAuthHeader() } : {},
    body: form,
  })
  if (!res.ok) {
    const err = await res.json().catch(() => ({}))
    throw new Error(err.error?.message || `chunk ${chunk.index} upload failed: ${res.status}`)
  }
  return chunk.index
}
```

## 四、后端接口约定

### 4.1 POST /api/upload/check（秒传 + 断点查询）

**请求体：**

```json
{
  "file_hash": "前端计算的文件 hash",
  "file_name": "原始文件名",
  "total_chunks": 10,
  "chunk_size": 2097152
}
```

**响应：**

- 文件已存在（已合并）：`{ "exists": true, "file_id": "xxx" }` → 前端秒传。
- 未合并，但有部分 chunk：`{ "exists": false, "uploaded_chunks": [0, 1, 2] }` → 前端只传缺失 chunk（断点续传）。

### 4.2 POST /api/upload/chunk（上传单块）

- `Content-Type: multipart/form-data`。
- 字段：`file_hash`、`chunk_index`、`total_chunks`、`file_name`、`chunk`（文件二进制）。
- 服务端按 `file_hash` 建目录，按 `chunk_index` 存文件（如 `chunks/{file_hash}/0`、`1`、…）。

### 4.3 POST /api/upload/complete（通知合并）

**请求体：**

```json
{
  "file_hash": "...",
  "file_name": "...",
  "total_chunks": 10,
  "chunk_size": 2097152
}
```

**响应：** `{ "job_id": "uuid" }`。后端异步合并，前端用 `job_id` 轮询状态。

### 4.4 GET /api/upload/status?job_id=xxx（轮询合并结果）

**响应：**

- `{ "status": "pending" | "merging" | "success" | "failed", "file_id": "...", "error": "..." }`
- `success` 时前端保存 `file_id` 并写入 localStorage，完成秒传/刷新恢复的数据源。

## 五、刷新后恢复任务

- **数据来源**：`listPendingTasks()` 扫描所有 `upload_task_*`，筛出 `status === 'waiting' | 'uploading'`。
- **恢复方式**：  
  - 若仅做「进度展示」：用持久化的 `totalChunks`、`uploadedChunks` 算进度条即可。  
  - 若需「继续传」：用户再次选择**同一文件**（同 hash），`runUploadTask` 会先 `getTaskState(fileHash)`，再调 check 得到 `uploaded_chunks`，只传缺失块，实现断点续传；若文件已合并，check 返回 `exists + file_id`，直接秒传。

注意：刷新后无法自动恢复「正在选择的 File 对象」，只能恢复「进度展示」或「用户重新选同一文件后继续传」。

## 七、小结

| 能力 | 实现方式 |
|------|----------|
| **切片上传** | 固定大小切片，每块单独 POST 到 `/api/upload/chunk`。 |
| **断点续传** | check 返回 `uploaded_chunks`，前端只传缺失块；每块完成写 localStorage。 |
| **秒传** | check 发现服务端已有该 `file_hash`（已合并）则返回 `file_id`，前端不再传 chunk。 |
| **进度持久化** | `upload_task_{fileHash}` 存 status、uploadedChunks、jobId、fileId 等。 |
| **刷新恢复** | `listPendingTasks()` 读未完成任务；用户重选同一文件后继续传或直接秒传。 |

## 八、「文件传给后端」是什么意思？后端存在哪？

### 8.1 什么叫「传给后端」

- **前端**：在浏览器里用 `fetch()` 把文件的**切片（chunk）**以 `multipart/form-data` 形式发到后端地址（如 `http://localhost:3001/api/upload/chunk`）。
- **后端**：是一台跑在你本机或服务器上的 **Python 服务**（本项目里是 `server_python/main.py`）。它收到 HTTP 请求后，把请求体里的二进制数据**写到本机磁盘**的某个目录里。

所以「文件传给后端」= 浏览器通过 HTTP 把文件内容发给你的 Python 服务，Python 把内容存到磁盘上的指定目录。

### 8.2 后端把文件存在哪里（本仓库）

后端用**当前工作目录**下的 `uploads` 作为根目录（可通过环境变量 `UPLOAD_ROOT` 覆盖）。  
若你在 `server_python` 目录下执行 `python main.py`，则「当前目录」是 `server_python`，根目录就是 **`server_python/uploads`**。

| 用途 | 路径（相对 server_python） | 说明 |
|------|----------------------------|------|
| 分片暂存 | `uploads/chunks/{file_hash}/0`、`1`、`2`… | 每个文件一个子目录，按 chunk 下标存文件；合并后会删除 |
| 合并临时文件 | `uploads/merged/` | 合并时的临时大文件，合并完成后会删掉（或若配置了阿里云则上传后删） |
| 已合并记录 | `uploads/merged_registry.json` | `file_hash -> file_id`，用于秒传、重启后识别已合并文件 |

**绝对路径示例（Windows 本机）：**

- 项目根：`e:\01\llm-chat-box2.0-main`
- 上传根目录：`e:\01\llm-chat-box2.0-main\server_python\uploads`
- 某文件的 chunk：`e:\01\llm-chat-box2.0-main\server_python\uploads\chunks\<file_hash>\0`、`1`、`2`…

若设置了 `UPLOAD_ROOT=D:\my_uploads`，则上述目录都会在 `D:\my_uploads` 下。

### 8.3 本地调试完整举例

**1）启动后端（在项目里用 Python 跑 FastAPI）**

```bash
cd e:\01\llm-chat-box2.0-main\server_python
.venv\Scripts\Activate.ps1   # 若有虚拟环境
python main.py
```

- 默认监听：`http://localhost:3001`
- 启动时会创建 `server_python/uploads`、`uploads/chunks`、`uploads/merged`（若不存在）。

**2）启动前端**

```bash
cd e:\01\llm-chat-box2.0-main
pnpm run dev
```

- 前端默认会请求 `http://localhost:3001`（见 `src/utils/api.js` 里 `BACKEND_BASE_URL`），所以会连到你刚启动的 Python 后端。

**3）在页面上传一个大文件**

- 在聊天/上传入口选一个**大于 2MB** 的文件（才会走分片），触发上传。
- 打开浏览器开发者工具 (F12) → Network，可以看到：
  - `POST .../api/upload/check`
  - 多次 `POST .../api/upload/chunk`
  - `POST .../api/upload/complete`
  - 多次 `GET .../api/upload/status?job_id=xxx`

**4）看后端「把文件存在哪」**

- 在资源管理器里打开：  
  `e:\01\llm-chat-box2.0-main\server_python\uploads\chunks`
- 会看到以「文件 hash」命名的子目录，里面是 `0`、`1`、`2`… 无扩展名的文件，就是按序的切片。
- 合并完成后，该 hash 目录会被删除，`uploads/merged_registry.json` 里会多一条 `file_hash -> file_id`。

**5）改存盘位置（可选）**

在 `server_python` 下建 `.env`，或在本机设置环境变量：

```env
UPLOAD_ROOT=D:/my_upload_folder
```

重启后端后，所有 chunk/merged/registry 都会在 `D:\my_upload_folder` 下，便于你单独备份或排查。
